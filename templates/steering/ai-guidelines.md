# AI 编程开发规范

> 本规范定义了一套结构化方法，旨在帮助 AI Agent 在长周期的软件开发任务中，能够跨越多个上下文窗口持续、稳定、安全地推进项目。

## 核心挑战：上下文断裂

长周期 AI 任务面临的最大问题是**上下文断裂**：
- **状态丢失**：每个新会话开始时，Agent 都是"失忆"的
- **过度冒进**：Agent 倾向于一次性完成所有功能，导致代码质量下降
- **过早完工**：Agent 看到已有代码，容易误判任务已完成
- **环境碎片**：未配置的环境或丢失的依赖导致新会话无法立即开始工作
- **回归盲区**：新功能破坏旧功能，Agent 毫无察觉地继续前进

## 核心原则

### 1. 上下文持久化

通过状态文件解决上下文断裂问题：
- **状态文件**：使用结构化文件（JSON/Markdown）记录项目状态
- **增量更新**：每次会话后更新状态，而非重写
- **交接日志**：记录决策、问题和下一步建议

### 2. 增量开发

**每次只完成一个功能**：
- ❌ 禁止一次开发多个功能
- ✅ 完成一个功能 → 测试 → 提交 → 更新状态 → 结束会话
- ✅ 粒度适中：一个功能 1-4 小时完成
- ✅ 独立可测：不依赖未完成的功能

### 3. 证据驱动

**必须提供工具执行结果作为完成证据**：
- ✅ 测试日志显示 PASS/✓
- ✅ E2E 测试截图或视频
- ✅ API 响应符合预期 schema
- ❌ "我认为功能已完成" - 缺乏工具证据

### 4. 安全执行

**命令白名单机制**：

#### ✅ 允许的命令

| 类别 | 命令 |
|:---|:---|
| 文件操作 | `ls`, `cat`, `head`, `tail`, `wc`, `grep`, `find`, `cp`, `mv` |
| Node.js | `npm`, `node`, `npx`, `yarn` |
| Python | `pip`, `python`, `pytest`, `black`, `flake8`, `mypy` |
| Go | `go`, `gofmt` |
| Rust | `cargo`, `rustc`, `rustfmt` |
| 版本控制 | `git` (all subcommands) |
| 进程管理 | `ps`, `lsof`, `sleep` |

#### ❌ 禁止的命令

| 类别 | 命令 | 原因 |
|:---|:---|:---|
| 提权 | `sudo`, `su` | 系统级风险 |
| 权限 | `chmod`, `chown` (除非明确必要) | 权限变更 |
| 破坏性 | `rm -rf /`, `mkfs`, `fdisk` | 数据不可恢复 |
| 网络后门 | `nc`, `netcat`, `telnet` | 安全隐患 |
| 防火墙 | `iptables`, `route` | 网络配置变更 |
| 盲目下载 | `curl \| bash`, `wget \| sh` | 不受审查的脚本 |
| 系统杀进程 | `kill -9` (系统进程) | 系统稳定性 |

### 5. 回归优先

**发现回归立即修复**：
1. 🛑 立即停止新功能开发
2. 🔧 优先修复回归问题
3. 标记受影响功能为 `"status": "regression"`
4. 📝 记录到 progress.log

## 开发工作流程

### Phase 1: 环境对齐与验证

1. **环境对齐**
   - 执行 `pwd`, `ls` 熟悉结构
   - 读取核心指南文档
   - 读取进度日志和功能清单
   - 查看 `git log --oneline -10` 了解最近变更

2. **环境验证**
   - 执行 `init.sh` 或现有测试，确保环境正常
   - 检查依赖是否安装（`node_modules/`, `venv/`, etc.）
   - 检查服务是否运行
   - 如果发现问题，**先修复再继续**

3. **回归检查**
   - 从已完成的功能中挑选 1-2 个核心功能
   - 运行其测试用例验证系统未被破坏
   - 如果有失败 → 立即修复，标记为 regression

### Phase 2: 任务选取

1. 从功能清单中挑选最高优先级的 `passes: false` 任务
2. 确保其所有 `dependencies` 均已完成
3. 将状态更新为 `"status": "in_progress"`

### Phase 3: 实现与验证

1. **执行实现**
   - 编写代码，**严禁**超出当前选定的任务范围
   - 遵循项目编码规范，添加必要注释
   - 编写相应的测试用例

2. **质量保证**
   - 使用工具验证功能
   - **证据驱动**：必须满足验证要求
   - 只有当所有测试用例状态为 `passed` 且满足完成标准时，方可标记为完成

### Phase 4: 持久化与交接

1. **Git 提交**
   - 执行 `git add` / `git commit`
   - 备注信息需详尽
   - 每个功能一个提交

2. **更新状态文件**
   - 更新功能清单中的状态
   - 在进度日志中留下交接说明

## 状态管理文件

### 功能清单 (feature_list.json)

功能的单一事实来源，每个功能必须包含：
- `id`, `category`, `description`, `priority`, `core`
- `status`, `dependencies`, `steps`, `test_cases`, `security_checks`, `passes`
- `validation_requirements`：定义完成功能所需的具体工具证据
- `completion_criteria`：原子化的完成清单
- `retry_count`, `max_retries`：自动错误恢复机制配置

### 进度日志 (progress.log)

面向人类/Agent 的自然语言进度总结：
- 采用增量追加模式
- 记录"已完成"、"进行中"、"待办事项"
- 包含下一步交接说明

### 架构文档 (architecture.md)

记录项目架构、技术栈选择和核心数据流。

### 环境初始化脚本 (init.sh)

脚本化环境配置：
- 运行此脚本后，任何 Agent 应该能立即执行测试或启动开发
- 必须是**幂等的**（可安全多次运行）

## 代码质量标准

### Git 提交规范

每次功能完成**必须**提交，**严禁**一次提交多个功能。

提交信息格式：
```
<type>(<scope>): <description> [Closes #feature-id]

- 实现内容 1
- 实现内容 2
- 添加测试用例
```

**Type 类型**：
- `feat:` 新功能
- `fix:` Bug 修复
- `refactor:` 代码重构
- `test:` 测试相关
- `docs:` 文档更新
- `chore:` 构建/工具相关

### 测试覆盖要求

| 项目类型 | 测试要求 |
|:---|:---|
| Web 项目 | 必须使用 Playwright/Cypress 进行 E2E 测试 |
| API 项目 | 验证响应状态码、数据格式、错误处理 |
| CLI 项目 | 测试命令行输入/输出、退出码、异常参数 |
| 通用 | 测试覆盖率 ≥ 70% |

### 代码审查清单

完成功能前必须验证：

**功能正确性**
- [ ] 所有验收标准已满足
- [ ] 所有测试用例通过并有工具证据
- [ ] 边界情况已处理
- [ ] 错误消息清晰有帮助

**代码质量**
- [ ] 遵循项目编码规范
- [ ] 无 TODO/FIXME/HACK 注释未跟踪
- [ ] 无死代码或未使用的导入
- [ ] Lint 零错误

**安全检查**
- [ ] 用户输入已验证和清理
- [ ] 无硬编码的秘密或凭据
- [ ] 敏感数据不以明文记录
- [ ] 认证/授权检查已到位

## 错误恢复机制

### 自动错误分类

| 错误类型 | 检测方法 | 处理方式 |
|:---|:---|:---|
| 环境 | 缺少 node_modules, venv 等 | 自动运行 init.sh |
| 依赖 | Module not found, ImportError | 自动安装缺失依赖 |
| 代码 | SyntaxError, TypeError | 自动修复，最多 3 次重试 |
| 需求不清 | 功能描述模糊 | 自动从文档中澄清 |
| 超出能力 | 外部 API 错误、权限问题 | 标记为 blocked，跳过 |

### 重试协议

当达到最大重试次数时：
1. 执行 `git reset --hard` 到上一个稳定状态
2. 将状态标记为 `"blocked"`
3. 记录具体的 `blocked_reason`
4. 跳过该任务，尝试队列中的下一个任务

## 项目完成标准

| 维度 | 要求 |
|:---|:---|
| 功能完成 | 所有功能 `passes: true`，无 `blocked` 或 `regression` |
| 测试覆盖 | 测试覆盖率 ≥ 70% |
| 代码质量 | 无 lint 错误，通过类型检查 |
| 文档 | README 完整，API 文档清晰，代码注释充分 |
| Git 历史 | 每个功能一个提交，消息清晰，无冗余 |

## 最佳实践

### 功能拆分
- **粒度适中**：一个功能 1-4 小时完成
- **独立可测**：不依赖未完成的功能
- **价值明确**：每个功能有清晰的业务价值
- **边界清晰**：功能之间职责分明

### 状态管理
- **及时更新**：完成功能后立即更新状态
- **人性化日志**：增量记录每个会话的决策与成果
- **定期回顾**：检查进度和剩余工作

### 代码质量
- **测试驱动**：先写测试，再写实现
- **持续重构**：保持代码整洁
- **文档同步**：代码和文档保持一致
- **版本控制**：小步提交，频繁提交

### 文档同步维护

**当需求或任务发生变更时，必须检查和更新文档**：

1. **识别变更范围**
   - 检测功能需求变更、任务优先级调整、依赖关系变化
   - 识别技术栈、架构决策的修改
   - 发现已删除或废弃的功能

2. **执行文档审查**
   - 检查所有相关文档是否与当前代码状态一致
   - 标识过时的文档内容
   - 确认文档引用的代码、API、配置是否仍然有效

3. **更新过时内容**
   - 更新 README.md 反映最新的项目说明
   - 更新架构文档记录新的设计决策
   - 更新 API 文档同步接口变更
   - 更新配置文件说明
   - 更新进度日志记录变更原因

4. **删除过时文档**
   - 删除已废弃功能的文档
   - 删除不再使用的配置说明
   - 删除过时的迁移指南
   - 在删除前确认无其他文档引用

5. **变更检查清单**
   ```
   每次需求或任务变更时：
   - [ ] README.md 是否需要更新？
   - [ ] 架构文档是否需要修改？
   - [ ] API 文档是否需要同步？
   - [ ] 是否有过时文档需要删除？
   - [ ] 进度日志是否记录了变更原因？
   - [ ] 相关注释是否需要更新？
   ```

**文档一致性原则**：
- 🔄 **实时同步**：代码变更后立即更新相关文档
- 🗑️ **及时清理**：废弃功能文档应立即删除
- 📝 **原因追溯**：重要变更应在进度日志中说明原因
- 🔍 **引用检查**：删除文档前检查是否被其他文档引用

## 绝对禁令

以下行为**严格禁止**：

### 🚫 禁止开发多个功能
- ❌ 完成 F001 后，自发开始 F002
- ✅ 完成 F001 → 提交 → 更新进度 → 结束会话

### 🚫 禁止跳过测试
- ❌ "这个改动很小，不需要测试"
- ✅ 任何代码改动都运行相关测试

### 🚫 禁止修改已完成的功能
- ❌ 发现 F001 有小问题，顺便修复
- ✅ 创建新的修复任务 F-BUG-001

### 🚫 禁止假设环境状态
- ❌ "假设数据库已创建"
- ✅ 运行 init.sh 验证环境

### 🚫 禁止无工具证据标记完成
- ❌ "我认为功能已完成"
- ✅ 提供工具执行结果（测试日志、截图、API 响应）

### 🚫 禁止使用禁止命令
- ❌ 使用 `sudo`, `rm -rf /`, `curl | bash` 等
- ✅ 仅使用白名单中的命令

## 违规后果

如果检测到违规：
1. 🛑 立即停止当前工作
2. 🔧 恢复到上一个 Git 提交
3. 📝 在 progress.log 中记录违规详情
4. ⚠️ 将功能状态标记为 "blocked"

**记住**：你的目标是高质量、可持续、安全地完成项目开发。遵循规范，减少犯错，确保每个功能都经过充分验证。
